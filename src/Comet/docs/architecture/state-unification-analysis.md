# Comet State Tracking Unification ‚ÄĒ Technical Analysis & Proposal

**Author:** Naomi (Source Generator Dev)  
**Date:** 2026-03-16  
**Requested by:** David Ortinau  
**Status:** Proposal

---

## Executive Summary

Comet currently runs two independent state tracking systems in parallel: **System A** (StateManager/Binding ‚ÄĒ a fine-grained, property-level change tracker using `INotifyPropertyRead` events) and **System B** (ReactiveScope ‚ÄĒ a coarse-grained, scope-based dependency tracker using `IReactiveSource` subscriptions). These systems share no common interface, are bridged by a 29-line `Suppress()/Resume()` hack in `ReactiveScope`, and impose a cognitive and maintenance burden disproportionate to a framework with zero external users. After reading all relevant source files, counting touch points (35 files for Binding, 17 for StateManager, 9 for ReactiveScope), analyzing the source generator templates, and evaluating four unification options against DX, performance, hot reload, and migration cost, **I recommend Option C: a unified reactive system** that extends the ReactiveScope/Signal infrastructure to handle fine-grained property updates. This eliminates ~1,200 lines of legacy code (StateManager + BindingObject + Binding), simplifies the source generator output, produces a cleaner developer API, and improves hot reload reliability ‚ÄĒ while preserving the fine-grained update performance that makes Comet faster than pure-rebuild frameworks like MauiReactor.

## 1. Current Architecture Deep Dive

### System A: StateManager/Binding (Fine-Grained, Legacy)

**Files:** `StateManager.cs` (593 lines), `Binding.cs` (424 lines), `BindingObject.cs` (197 lines), `State.cs` (106 lines)  
**Total:** ~1,320 lines of core infrastructure

**How it works:**

1. `State<T>` extends `BindingObject`, which implements `INotifyPropertyRead`.
2. When `State<T>.Value` is read, `CallPropertyRead()` fires, which calls `StateManager.OnPropertyRead()`.
3. `StateManager` maintains thread-static lists (`_currentReadProperties`) that accumulate all reads during a "building" phase.
4. When a `Binding<T>` is constructed with a `Func<T>`, `ProcessGetFunc()` brackets the evaluation with `StartProperty()`/`EndProperty()` to capture which `INotifyPropertyRead` objects were read.
5. `Binding<T>.BindToProperty()` registers these tracked properties into `BindingState.ViewUpdateProperties` ‚ÄĒ a `Dictionary<(INotifyPropertyRead, string), HashSet<(string, Binding)>>`.
6. When a property changes, `StateManager.OnPropertyChanged()` dispatches to all views via `NotifyToViewMappings`, which calls `View.BindingPropertyChanged()`.
7. `BindingState.UpdateValue()` finds the specific binding(s) for that property and calls `Binding.BindingValueChanged()`.
8. The binding re-evaluates its `Func<T>`, gets the new value, and calls `View.ViewPropertyChanged()` which updates the native handler for just that one property.

**Key complexity points:**
- `StateManager` holds 7 static dictionaries protected by a `ReaderWriterLockSlim`
- Thread-static stacks track which View is "currently building"
- The implicit operator `Binding<T>(T value)` requires `StateManager.EndProperty()` to distinguish data-bound values from constants ‚ÄĒ this is action-at-a-distance that confuses developers
- `BindingState` has separate `GlobalProperties` (trigger full rebuild) vs `ViewUpdateProperties` (trigger fine-grained update) ‚ÄĒ a two-tier dispatch that adds branching
- Batching is bolted on via `BeginBatch()`/`EndBatch()` with `_dirtyBindings` and `_viewsNeedingReload` lists

**Example ‚ÄĒ generated control (from Mustache template):**
```csharp
// Generated by CometViewSourceGenerator
public partial class Slider : View {
    Binding<double> value;
    public Binding<double> Value {
        get => value;
        private set => this.SetBindingValue(ref this.value, value);
    }
    
    public Slider(Binding<double> value = default, ...) {
        Value = value;  // triggers BindToProperty ‚Üí StateManager registration
    }
}
```

**Example ‚ÄĒ user code:**
```csharp
readonly State<double> progress = 0.5;

new Slider(
    value: () => progress.Value,     // Binding<double> via implicit Func<T> conversion
    minimum: 0.0,
    maximum: 1.0
)
```

### System B: ReactiveScope (Coarse-Grained, New)

**Files:** `ReactiveScope.cs` (69 lines), `Signal.cs` (94 lines), `Computed.cs` (129 lines), `Effect.cs` (88 lines), `ReactiveScheduler.cs` (212 lines), `SubscriberList.cs` (85 lines), `IReactiveSource.cs` (7 lines), `IReactiveSubscriber.cs` (5 lines), `ReactiveEnvironment.cs` (~60 lines), `ReactiveDiagnostics.cs` (~30 lines), `SignalList.cs` (~50 lines)  
**Total:** ~999 lines in `src/Comet/Reactive/`

**How it works:**

1. `Signal<T>` (and `Reactive<T>`, `Computed<T>`) implement `IReactiveSource`.
2. When `Body()` is invoked, `GetRenderViewReactive()` wraps the call in `ReactiveScope.BeginTracking()`.
3. Any `Signal<T>.Value` read within the scope calls `ReactiveScope.Current.TrackRead(this)`.
4. After `Body()` returns, `EndTracking()` returns the set of `IReactiveSource` objects read.
5. A `BodyDependencySubscriber` subscribes to each tracked source.
6. When a signal changes, `SubscriberList.NotifyAll()` ‚Üí `BodyDependencySubscriber.OnDependencyChanged()` ‚Üí `ReactiveScheduler.MarkViewDirty()`.
7. `ReactiveScheduler.EnsureFlushScheduled()` dispatches to the UI thread, calls `View.Reload()`, which re-executes `Body()`, diffs old and new view trees, and transfers handlers.

**Key elegance points:**
- `ReactiveScope` is 69 lines ‚ÄĒ dead simple
- No static dictionaries, no lock contention, no thread-static stacks
- `SubscriberList` uses `WeakReference<T>` ‚ÄĒ automatic cleanup, no manual `Disposing()` call needed
- `ReactiveScheduler` batches multiple signal changes into one flush cycle automatically
- The Signal ‚Üí Computed ‚Üí Effect DAG is a proper reactive graph with version tracking and cycle detection

**Example ‚ÄĒ user code:**
```csharp
readonly Signal<int> selectedIndex = 0;

[Body]
View body() => new VStack {
    new Text($"Selected: {selectedIndex.Value}"),  // tracked by ReactiveScope
    new Button("Next", () => selectedIndex.Value++)
};
// When selectedIndex changes ‚Üí body() re-executes ‚Üí diff ‚Üí handler reuse
```

### The Bridge: Suppress/Resume

In `Binding<T>.ProcessGetFunc()` (line 108):
```csharp
var scope = ReactiveScope.Suppress();    // null out ReactiveScope._current
var result = Get.Invoke();                // reads go to StateManager only
ReactiveScope.Resume(scope);             // restore ReactiveScope._current
```

This prevents double-tracking: without it, a `State<T>.Value` read inside a `Binding<T>` Func would register in BOTH `StateManager._currentReadProperties` (via `CallPropertyRead` ‚Üí `StateManager.OnPropertyRead`) AND `ReactiveScope._current._reads` (via... well, `State<T>` doesn't implement `IReactiveSource`, but `Reactive<T>` does, and `Signal<T>` does). The bridge is necessary because the two systems use different notification interfaces on the same value read.

### The Dual-Nature Problem: `Signal<T>`

`Signal<T>` implements BOTH interfaces:
- `IReactiveSource` ‚ÄĒ for ReactiveScope tracking
- `INotifyPropertyRead` ‚ÄĒ for StateManager compatibility

In `Signal<T>.Value` getter (line 33-38):
```csharp
get {
    if (ReactiveScope.Current != null)
        ReactiveScope.Current.TrackRead(this);  // reactive path
    else
        PropertyRead?.Invoke(this, ValueChangedArgs);  // legacy path
    return _box.Value;
}
```

This `if/else` branch is the fundamental tension: Signal must serve two masters.

### Where Each System Is Used Today

| Area | System A (StateManager/Binding) | System B (ReactiveScope) |
|------|------|------|
| Generated controls (19 controls) | ‚úÖ Constructor params are `Binding<T>` | ‚ĚĆ Not used |
| Handwritten controls (Picker, Image, etc.) | ‚úÖ Manual `Binding<T>` properties | ‚ĚĆ Not used |
| Extension methods (fluent API) | ‚úÖ `SetEnvironment` takes `Binding<T>` | ‚ĚĆ Not used |
| View.Body() dependency tracking | ‚ĚĆ Uses `StateBuilder` for view stack only | ‚úÖ `GetRenderViewReactive()` |
| Component.SetState() | ‚úÖ Uses `StateManager.BeginBatch()/EndBatch()` | ‚ĚĆ Not used |
| Signal/Computed/Effect | Partial (via `INotifyPropertyRead`) | ‚úÖ Primary system |
| State<T> | ‚úÖ Primary holder | ‚ĚĆ Not reactive |
| Reactive<T> | ‚úÖ Inherits from State<T> | ‚úÖ Implements IReactiveSource |
| AutoNotify generator | ‚úÖ Generates `StateManager.OnPropertyRead/Changed` | ‚ĚĆ Not used |
| StateWrapper generator | ‚úÖ Generates `StateManager.OnPropertyChanged` | ‚ĚĆ Not used |
| Source generator (CometViewSourceGenerator) | ‚úÖ All templates use `Binding<T>` | ‚ĚĆ Not used |

## 2. The Four Options

### Option A: Pure ReactiveScope (Remove StateManager/Binding Entirely)

**Concept:** All state changes go through body rebuild + smart diff. Controls don't have `Binding<T>` properties; they take static values. When any tracked signal changes, the view's `Body()` re-executes and the diff algorithm figures out what actually changed at the handler level.

**Developer API:**
```csharp
readonly Signal<string> name = "World";

[Body]
View body() => new VStack {
    new Text($"Hello, {name.Value}!"),
    new TextField(name.Value, onChanged: v => name.Value = v),
    new Slider(progress.Value, min: 0, max: 1, onChanged: v => progress.Value = v),
};
```

**What changes:**
- Generated controls take `T` instead of `Binding<T>` (e.g., `TextField(string text, ...)` not `TextField(Binding<string> text, ...)`)
- No more `Binding<T>`, `BindingObject`, `BindingState`, `StateManager`
- All updates go through `ReactiveScheduler` ‚Üí `View.Reload()` ‚Üí `GetRenderViewReactive()` ‚Üí `Diff()`
- Two-way binding handled via explicit `onChanged` callbacks (like React/SwiftUI)
- Source generator templates simplified: no `SetBindingValue`, no binding registration

**Performance implications:**
- Single property change (e.g., slider thumb drag): Instead of `Binding.EvaluateAndNotify() ‚Üí ViewPropertyChanged() ‚Üí handler.UpdateValue()`, we get `ReactiveScheduler.MarkViewDirty() ‚Üí Reload() ‚Üí Body() ‚Üí Diff() ‚Üí handler transfer`. This is significantly more work per tick.
- Page navigation: Similar cost (body rebuild either way).
- List scrolling: Same.
- Animation: Potentially problematic ‚ÄĒ 60fps slider or scroll needs per-frame body rebuild.

**Hot reload impact:** Simpler ‚ÄĒ no binding state to preserve/transfer. Hot reload just re-executes `Body()` and diffs.

**Source generator impact:** Major simplification. Templates remove `Binding<T>` parameters, `SetBindingValue`, `BindToProperty`. Controls become plain value holders.

**Risk:** Performance regression for high-frequency updates. MauiReactor uses this model and compensates by being very aggressive about minimizing diff cost, but Comet's diff is more complex (it supports hot reload state transfer).

### Option B: Pure StateManager (Remove ReactiveScope)

**Concept:** Remove the reactive system entirely. All dependency tracking goes through `StateManager.OnPropertyRead()` / `OnPropertyChanged()`. Body-level rebuilds triggered by `GlobalProperties` in `BindingState`.

**Developer API:**
```csharp
readonly State<string> name = "World";

[Body]
View body() => new VStack {
    new Text(() => $"Hello, {name.Value}!"),     // Binding<string> via Func
    new TextField(name, onChanged: v => name.Value = v),
};
```

**What changes:**
- Remove `Reactive/` directory entirely (999 lines)
- Remove `Signal<T>`, `Computed<T>`, `Effect`, `ReactiveScheduler`
- `GetRenderViewReactive()` replaced with the old `StateBuilder` pattern
- Body-level dependencies tracked via `BindingState.GlobalProperties`

**Performance implications:**
- Fine-grained updates remain fast.
- Body-level dependencies become "global" state ‚ÄĒ any change triggers full rebuild. No targeted body-dependency tracking.
- No automatic batching (ReactiveScheduler batches multiple signal changes into one flush).

**Hot reload impact:** More complex ‚ÄĒ must preserve `BindingState`, re-register all bindings after type replacement.

**Source generator impact:** No change (already uses Binding<T>).

**Risk:** Loses the entire reactive primitive library (Signal, Computed, Effect) which is the forward-looking API. `State<T>` is already marked `[Obsolete]`. This is a regression to the old architecture.

**Verdict:** This option moves backwards. The reactive system is clearly the better-designed system. Not recommended.

### Option C: Unified Reactive System (Recommended)

**Concept:** Extend the ReactiveScope/Signal infrastructure to handle fine-grained property updates. `Signal<T>` becomes the one true state primitive. The source generator produces controls that subscribe to individual `IReactiveSource` instances for their properties, avoiding full body rebuilds for single-property changes.

**Key insight:** `ReactiveScope` already does what `StateManager.StartProperty()/EndProperty()` does ‚ÄĒ it tracks which sources were read during a function evaluation. The difference is that `StateManager` associates those reads with a `Binding` and registers them for targeted notification, while `ReactiveScope` associates them with a view body and triggers a full rebuild.

**The unified approach:** Give each control property its own lightweight `ReactiveScope` subscription, similar to how `Computed<T>` works. When a signal changes, only the specific property subscriber re-evaluates, exactly like `Binding<T>` does today but using the reactive infrastructure.

**Developer API:**
```csharp
readonly Signal<string> name = "World";
readonly Signal<double> progress = 0.5;

[Body]
View body() => new VStack {
    // Static value ‚ÄĒ no tracking, no overhead
    new Text("Welcome"),
    
    // Func ‚ÄĒ creates a PropertySubscription under the hood  
    // Fine-grained: only this text property re-evaluates when name changes
    new Text(() => $"Hello, {name.Value}!"),
    
    // Signal directly ‚ÄĒ creates a bidirectional PropertySubscription
    new TextField(name),
    
    // Func for computed display
    new Slider(() => progress.Value, min: 0, max: 1, 
               onChanged: v => progress.Value = v),
};
```

**What changes internally:**

Replace `Binding<T>` with `PropertySubscription<T>` built on `IReactiveSource`:

```csharp
// New: replaces Binding<T>
internal sealed class PropertySubscription<T> : IReactiveSubscriber, IDisposable
{
    readonly Func<T>? _compute;
    HashSet<IReactiveSource>? _dependencies;
    T _currentValue;
    View? _view;
    string? _propertyName;
    
    // For static values
    public PropertySubscription(T value) { _currentValue = value; }
    
    // For reactive expressions
    public PropertySubscription(Func<T> compute) 
    {
        _compute = compute;
        Evaluate();  // uses ReactiveScope.BeginTracking() internally
    }
    
    // For Signal<T> directly (bidirectional)
    public PropertySubscription(Signal<T> signal)
    {
        _compute = () => signal.Value;
        Evaluate();
    }
    
    void Evaluate()
    {
        using var scope = ReactiveScope.BeginTracking();
        _currentValue = _compute!();
        var newDeps = scope.EndTracking();
        // Update subscriptions (same pattern as Computed<T>)
        UpdateDependencies(newDeps);
    }
    
    public void OnDependencyChanged(IReactiveSource source)
    {
        var oldValue = _currentValue;
        Evaluate();
        if (!EqualityComparer<T>.Default.Equals(oldValue, _currentValue))
            _view?.ViewPropertyChanged(_propertyName, _currentValue);
    }
}
```

**Generated control code would become:**
```csharp
public partial class Slider : View {
    PropertySubscription<double> _value;
    public PropertySubscription<double> Value {
        get => _value;
        private set => SetPropertySubscription(ref _value, value, nameof(Value));
    }
    
    // Static value overload
    public Slider(double value = default, ...) { ... }
    
    // Func overload ‚ÄĒ fine-grained reactive
    public Slider(Func<double> value = default, ...) { ... }
    
    // Signal overload ‚ÄĒ bidirectional
    public Slider(Signal<double> value = default, ...) { ... }
}
```

**What gets deleted:**
- `StateManager.cs` (593 lines)
- `BindingObject.cs` (197 lines) ‚ÄĒ `INotifyPropertyRead` interface no longer needed
- `Binding.cs` (424 lines) ‚ÄĒ replaced by `PropertySubscription<T>`
- `State.cs` (106 lines) ‚ÄĒ replaced by `Signal<T>`
- `Reactive.cs` (65 lines) ‚ÄĒ bridge class no longer needed
- `ReactiveScope.Suppress()/Resume()` (29 lines) ‚ÄĒ no more dual tracking
- `AutoNotifyGenerator` ‚ÄĒ regenerate to use reactive primitives
- `StateWrapperGenerator` ‚ÄĒ regenerate to use reactive primitives
- The `[Obsolete]` `State<T>` class entirely

**What gets kept/extended:**
- `ReactiveScope.cs` ‚ÄĒ used by both body tracking AND property subscriptions
- `Signal<T>` ‚ÄĒ primary state primitive (drops `INotifyPropertyRead` implementation)
- `Computed<T>` ‚ÄĒ derived values
- `Effect` ‚ÄĒ side effects
- `ReactiveScheduler` ‚ÄĒ batching and flush scheduling
- `SubscriberList` ‚ÄĒ subscriber management

**Performance implications:**
- **Single property update (slider drag):** `Signal.Value` set ‚Üí `SubscriberList.NotifyAll()` ‚Üí `PropertySubscription.OnDependencyChanged()` ‚Üí `Evaluate()` ‚Üí `ViewPropertyChanged()` ‚Üí handler update. This is the SAME number of hops as the Binding path today, but without `StateManager`'s lock contention and dictionary lookups.
- **Body rebuild:** Same as today ‚ÄĒ `BodyDependencySubscriber.OnDependencyChanged()` ‚Üí `ReactiveScheduler.MarkViewDirty()` ‚Üí `Reload()`.
- **Batching:** `ReactiveScheduler` already batches. Multiple signal changes ‚Üí single flush ‚Üí each `PropertySubscription` evaluates once ‚Üí each handler updates once. This replaces `StateManager.BeginBatch()/EndBatch()`.
- **Animation:** PropertySubscription evaluations are cheaper than Binding evaluations because there's no `StateManager.StartProperty()/EndProperty()` overhead.

**Hot reload impact:**
- Simpler. No `BindingState` to preserve. `PropertySubscription` instances are recreated when `Body()` re-executes (same as Computed dependencies).
- `ReactiveScope`-based tracking means hot reload just needs to re-execute `Body()` and let subscriptions re-establish naturally.

**Source generator impact:**
- Templates change from `Binding<T>` ‚Üí `PropertySubscription<T>`.
- Constructor overloads simplify: `Signal<T>`, `Func<T>`, and static `T` overloads (no implicit conversion magic).
- `SetBindingValue` ‚Üí `SetPropertySubscription` (similar pattern, simpler internals).
- Extension methods change from `Binding<T>` ‚Üí the new type (but the `SetEnvironment` pattern stays the same).

### Option D: Status Quo (Keep Both)

**Concept:** Leave both systems in place with the Suppress/Resume bridge.

**Pros:**
- Zero migration cost.
- Both systems work (mostly).

**Cons:**
- Every new developer must understand BOTH systems.
- `Signal<T>` must implement both `IReactiveSource` AND `INotifyPropertyRead` ‚ÄĒ the dual-path if/else in the getter.
- `State<T>` is `[Obsolete]` but `Binding<T>` depends on it via `BindingObject`.
- Adding new features (e.g., fine-grained environment tracking) requires touching BOTH systems.
- The `Suppress()/Resume()` bridge is fragile ‚ÄĒ if someone introduces an async body evaluation, the thread-static bridge breaks.
- Ongoing maintenance of 1,320 lines of legacy code that parallels 999 lines of modern code.

### DX Deep Dive

**Option A** has the simplest API (just values and callbacks) but loses the ability to do fine-grained updates without a full body rebuild. Users would be confused when a slider drag causes a visible re-render stutter.

**Option C** has a slightly more complex API (you need to understand `Signal<T>`, `Func<T>`, and static values) but the complexity maps to real performance benefits. The three overloads are natural:
- Static `T` ‚Üí no reactivity (fastest)
- `Func<T>` ‚Üí fine-grained reactive (per-property updates)
- `Signal<T>` ‚Üí bidirectional fine-grained reactive (input controls)

This is similar to SwiftUI's `@State`/`@Binding` distinction but with less annotation magic.

### Performance Deep Dive

The critical benchmark is **slider thumb drag at 60fps**. Each frame fires `onChanged` which sets `Signal.Value`, which must update the slider's displayed value AND any text label showing the numeric value.

**Option A (rebuild):** Each frame ‚Üí `body()` re-executes ‚Üí entire VStack allocates ‚Üí Diff compares every child ‚Üí finds Slider unchanged (same value it just set) ‚Üí finds Text changed ‚Üí updates Text handler. Cost: O(N) where N = number of children in body.

**Option C (fine-grained):** Each frame ‚Üí `Signal.Value` set ‚Üí `PropertySubscription` for Slider value re-evaluates ‚Üí handler updates slider. `PropertySubscription` for Text Func re-evaluates ‚Üí handler updates text. Cost: O(K) where K = number of bindings to that signal (typically 1-3).

For a page with 50 controls where only 2 depend on the slider signal, Option C is ~25x less work per frame.

### Hot Reload Deep Dive

**System A's hot reload problem:** When a type is replaced, all `BindingState` registrations (the `ViewUpdateProperties` dictionary) reference the OLD view instance. `TransferState()` copies `ChangedProperties` but doesn't re-register bindings. If the new body creates different bindings, the old registrations become stale. This is a known source of hot reload flakiness.

**System B's hot reload:** ReactiveScope subscriptions are rebuilt every time `Body()` executes. No stale state possible. When hot reload replaces a view type, `Body()` re-executes, new subscriptions are created, old ones are cleaned up by `WeakReference` GC.

**Option C** inherits System B's clean hot reload story. `PropertySubscription` instances are recreated when controls are reconstructed during `Body()`, and their `IReactiveSubscriber` subscriptions re-establish naturally via `ReactiveScope.BeginTracking()`.

## 4. Recommendation

**I recommend Option C: Unified Reactive System.**

### Rationale

1. **Zero users = zero breaking-change cost.** The migration cost is purely internal engineering effort, not ecosystem disruption. This is the perfect moment to fix a foundational flaw.

2. **Best of both worlds.** Option C preserves fine-grained property updates (Comet's advantage over MauiReactor's pure-rebuild model) while using the cleaner reactive infrastructure (Comet's advantage over the old StateManager design).

3. **Competitive positioning.** MauiReactor does full body rebuilds ‚ÄĒ Comet with unified fine-grained reactivity would be objectively faster for property-heavy UIs (sliders, text fields, animations). SwiftUI uses fine-grained tracking ‚ÄĒ Comet would be architecturally similar but with a simpler, more explicit API.

4. **Maintenance sustainability.** Maintaining two parallel systems is unsustainable. Every new feature (theming, accessibility, animations) must be wired through both. A unified system halves the integration surface.

5. **Hot reload reliability.** The reactive subscription model is inherently more hot-reload-friendly than the static dictionary model. Given that hot reload is a key DX differentiator for Comet, this matters.

6. **Source generator simplification.** The current generator templates have 13 references to `Binding<T>`. Replacing with `PropertySubscription<T>` (or whatever we name it) using consistent `ReactiveScope` tracking eliminates the need for `StateManager.StartProperty()/EndProperty()` calls in generated code.

### What This Looks Like for Framework Users

**Before (current):**
```csharp
// Must understand State<T>, Binding<T>, implicit conversions, 
// when Func is needed vs static value, [Obsolete] warnings
readonly State<string> name = "World";

[Body]
View body() => new VStack {
    new Text(() => $"Hello, {name.Value}!"),
    new TextField(() => name.Value, v => name.Value = v),
};
```

**After (unified):**
```csharp
// One primitive: Signal<T>. Clear overloads.
readonly Signal<string> name = "World";

[Body]
View body() => new VStack {
    Text(() => $"Hello, {name.Value}!"),
    TextField(name),  // Signal<T> overload = bidirectional
};
```

The second form is cleaner because:
- `Signal<T>` is the only state type to learn
- `TextField(name)` with a `Signal<T>` overload makes bidirectional binding obvious
- `Text(() => ...)` with a `Func<T>` overload makes computed display obvious
- No `Binding<T>`, no implicit conversions, no `[Obsolete]` warnings

## 5. Migration Plan

### Phase 1: PropertySubscription Core (Size: M ‚ÄĒ ~1 week)

**Goal:** Build `PropertySubscription<T>` as a new type that uses `ReactiveScope` for dependency tracking, side-by-side with existing `Binding<T>`.

**Tasks:**
